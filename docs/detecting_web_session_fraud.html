Detecting Intruders Through Website Session Tracking¶

by Kavin Manickaraj

Part 0. Problem Statement¶

Storing a consumer's behavioral data is quickly becoming standard operating procedure as a growing range of industries look to provide users with customized experiences. Ranging from banking[1] and transportation[2] to even fashion[3] and food[4], businesses are relying on this data to improve their services. But as they continue to do so, it becomes increasingly important to determine if unauthorized activity ever occurs. Whether it's catching fraudulent credit card purchases or flagging incendiary social media bots, there are few who won't benefit from this technology.

In this article, we will explore one of the more obvious applications of this strategy by attempting to categorize web sessions between those of an authorized user surfing the web and those of a nefarious third-party hoping to sway customized results (such as ads and suggested content) away from the user's interests. Since the data used in this project is purely for test purposes, let's look at how different modeling classifiers perform with this task, and how feature engineering may help achieve better results. We will consider model optimization and hyperparameter tuning in a later article.

To define our strategy more concretely, we will perform the following steps:

  1. Framing the problem - you are here!
  2. Data pre-processing - we import and clean the raw data of errors, missing values, and unhelpful formats
  3. Exploratory data analysis - we identify trends, select features, and create our training datasets
  4. Modeling and testing - we weigh the benefits (and detriments) of various modeling strategies

As an additional note, this data was obtained from this Kaggle competition used in some online communities (such as mlcourse.ai) as a teaching tool for aspiring machine learning engineers. The creators have specifically asked us former participants not to publish any well-performing kernels and keep the challenge for future students. Therefore, this article will omit certain elements to respect their wishes.

Within each row of our data, we see a session_id followed by the websites visited during this session (listed by site ID) along with their respective timestamps. Each web session tracks at most ten sites, after which the data is truncated. For sessions with less than ten sites, null values are placed in unused columns. There is a maximum of 253,561 entries in each column. However, some columns show less entries indicating where null values are present. Moreover, the time columns are seen as the object type rather than the more useful datetime64.

  • Convert the time columns to datetime64
  • Fill NaN cells in the site columns with 0 (but skip this step for the time columns... more on this later)

Part 2: Exploratory Data Analysis¶

When we put data through a predictive machine learning model, whether it be logistic regression or a highly complex neural network, our end objective is always the same: to define the relationships between features that describe an occurrence and the outcome of that occurrence. In this way we want our "machines" to perform some cost analysis to determine how changes in the data affect our target state. However, this analysis is not without its trade-offs. As the complexity of these data trends increases, our algorithms require more time, processing power, and detailed structuring to build up a "knowledge" of both the low-level and high-level patterns.

By exploring some of the more apparent patterns ourselves, we can essentially "pre-digest" the data before passing it along to our models. In a way, this provides a learning boost and allows algorithms more resources to define the harder-to-detect dependencies.

The objective of this section is to define some of these higher-level trends without spending our models' algorithmic resources to do so. Let's begin by looking at the number of unique sites visited during all web sessions.

There are many sites that include "google" in the web address, but we don't know if it's characteristic of either authorized or unauthorized web sessions to have one of these pages in the session history. In fact, we would need to consider the site IDs in each of the site columns to find a potential pattern -- not to mention other web address keywords that might be more useful.

One way to help interpret this data is to encode the web address using some variation of the One-Hot Encoding technique. Essentially, if we can categorize instances of specific keywords/phrases present in a web address, our modeling algorithm may be able to find some underlying patterns involving those words and phrases. For this scenario, we want to use the term frequency-inverse document frequency(tf-idf) strategy to encode our web address information. By defining the n-grams (effectively, the number of words in a phrase) to be encoded, our model gains the ability to drop frequently used terms not unique to either user (like "google" and "com") while still considering more complex phrases (like "accounts google").

To start this, we first need to replace the site IDs in each "site" column with their corresponding web addresses. We will create a new dictionary that inverts the keys and values of our original site ID dictionary and replace the site IDs in each relevant column accordingly.

It's important to know that this vectorizer function converts our dense matrix into a sparse one. Our newly encoded matrix has roughly 250,000 entries and 10,000 columns. We can easily calculate how much memory a dense matrix of this size would require:

$$(250,000)\times(10,000)\times(8\ \text{bytes}) = 20\ \text{GB of RAM}$$

Since our data is mostly comprised of zeroes (as a result of encoding 10,000 unique n-grams), memory is conserved by taking the form of a sparse matrix.

Our next step is to add time-dependent features. To do this, let's create a new dataframe of some basic time data we want to visualize. The features we'd like to investigate are:

  • start hour
  • stop hour
  • start day
  • start month
  • year
  • total session minutes

We should scale these features so that the magnitude of the values does not influence how our models see the data. In other words, the year 2013 should not be more important than a 20 minute web session simply because it is two orders of magnitude larger.

In [19]:
def add_time_features(dataframe, scaler):
    ''' This function takes a dataframe and a scaler function (or a pipeline of transformers) 
    and outputs a dataframe of extracted time features as well as a transformed/scaled 
    dataframe defined by the scalers provided.'''
    
    df = pd.DataFrame(index=dataframe.index)
    t = 'time1'
    df['start_hour'] = dataframe[t].apply(lambda ts: ts.hour)
    df['stop_hour'] = dataframe[time_cols].max(axis=1).apply(lambda ts: ts.hour)
    df['start_day'] = dataframe[t].apply(lambda ts: ts.weekday())
    df['start_month'] = dataframe[t].apply(lambda ts: ts.month)
    df['year'] = dataframe[t].apply(lambda ts: ts.year)
    df['session_mins'] = round((dataframe[time_cols].max(axis=1) - 
                                dataframe[time_cols].min(axis=1)) 
                               /np.timedelta64(1, 'm'), 2)
    df_scaled = pd.DataFrame(scaler.fit_transform(df),
                             index=df.index,
                             columns=df.columns)
    return df, df_scaled

Looking at these plots, it's easy to tell that the data is not linearly separable -- that is we can't draw a continuous straight line to separate the data by user. We can engineer some of our existing features to offer our models more relationships to consider. In this scenario, let's consider second-order polynomial interpretations involving the following features: session_mins, start_hour, start_month, start_day.

Using the scaler PolynomialFeatures(degree=2, include_bias=False), we can create polynomial combinations of features with degree less than or equal to 2 without a bias term. For example, if we had features [a, b], the scaler would output [a, b, a2, a*b, b2]. In this way, our four features would output a total of 14 columns: 4x identity columns, 4x squared columns, and 6x interaction columns. Since we already have the four initial identity columns represented in our train_times dataframe, we can write a function that performs this polynomial operation but only outputs the 10 new data columns. Additionally, we want to normalize these results for the same reasons we did so earlier.

In [24]:
def add_poly_feats(dataframe):
    
    # define the features to be considered
    feat_list = ['session_mins', 'start_hour', 'start_month', 'start_day']
    
    # create a pipeline to use PolynomialFeatures() then normalize the results with StandardScaler()
    pipeline = Pipeline([('poly', PolynomialFeatures(2, include_bias=False)), ('sscaler', StandardScaler())])
    
    # build a new dataframe to hold the results and only return the last 10 columns
    values = pipeline.fit_transform(dataframe[feat_list])
    df_scaled = pd.DataFrame(values, index=dataframe.index)
    return df_scaled[list(range(4,len(df_scaled.columns)))]

def add_time_of_day_features(dataframe):
    '''Takes a dataframe, extracts the starting hour, and interprets the time of day.
     This function outputs a DataFrame with OHE vectorized data corresponding to the time of day defined as:
    "morning"
    "afternoon"
    "evening"
    
    We can skip a "night" classification because there is no web usage between 23:00 and 7:00
    '''
    
    df = pd.DataFrame(index=dataframe.index)
    hour = dataframe['time1'].apply(lambda ts: ts.hour)
    df['morning'] = ((hour >= 6) & (hour <= 11)).astype('int') # between 6:00 and 12:00
    df['afternoon'] = ((hour >= 12) & (hour <= 17)).astype('int') # between 12:00 and 18:00
    df['evening'] = ((hour >= 18) & (hour <= 23)).astype('int') # between 18:00 and 24:00
    
    df_labels = pd.DataFrame(index=dataframe.index, 
                             data=np.dot(df.values, np.array(df.columns)), 
                             columns=['TOD'])
    
    return df, df_labels

Part 3. Modeling and Testing¶

There are many techniques in the 'Classification Problem' toolbox, and there's no way to get to them all in one article. To give us a good sample of capabilities, let's look at these four popular strategies:

  1. Random Forests
  2. Logistic Regression
  3. Support Vector Machine
  4. XGBoost

3.1. Random Forest Classifier¶

First to bat is a random forest classifier. Essentially an ensemble of decision trees, each estimator in the random forest classifier takes in a bootstrapped set of the training data and randomly splits it to "weigh" how alike or different the two groups are. When it determines where the best split should occur, it branches the two groups and passes them to the next decision node. The classifiers then repeat this process until they either classify each of the examples or until some stop-point is reached. The forest then takes all decision trees produced by the estimators and applies the logic behind Condorcet's Jury Theorem to produce the best answer after weighing each tree's results.

One major advantage of this method is that consulting many different decision trees over various bootstrapped data keeps the forest classifier from overfitting. While it's still true that overfitting is possible given the right inputs, the default parameters for Scikit-learn's RandomForestClassifier() should give us a generally acceptable solution.

The biggest drawback to a random forest is processing time (at least compared to some of the other models we will discuss here). Depending on both the training set size and number of estimators used, random forests are not the slowest algorithm of the bunch...but they certainly aren't the fastest. This time penalty renders random forest ineffective for real-time problem solving. However, we can afford the time for our purposes.

Let's consider a model with 50 estimators and run all three of our datasets through it. But there's just one problem: once we train our model on the different datasets, we still need to test our model for accuracy. Furthermore, our issue of imbalanced data means we can't rely on our conventional definition of accuracy to test our model -- since the "authorized" sessions make up just less than 1% of the total dataset, we could just classify all sessions as "unauthorized" and still be correct 99% of the time. To work around this, we will use the ROC-AUC scoring metric to give additional importance to maximizing true-positives and minimizing false-positives. A more detailed explanation of this metric can be found in the Additional Info section below.

Finally, let's consider what data we will use when testing our models. We'll use cross-validation to hold out some percentage of our total data, train the models with the remaining portion, then test them with our hold-out data to get AUC values. Doing this a few times -- let's choose five for this problem -- and taking the average will give us an idea of how our models perform based on the different datasets. One important note is that our data may be time dependent so it's best we take this under consideration and use the TimeSeriesSplit() cross-validation generator. This preserves any time-dependencies during our cross-validation step.

We can write a function to calculate the cross-validation scores for each of the three data sets and output their respective scores in the order they're listed in the dictionary above:

In [38]:
def run_random_forest(train_dict, y_train, n_estimators=50):
    """Takes a dictionary of time-dependent datasets and associated target value (must be the same for all datasets) 
    and outputs an average cross-validation score from a random forest classifier. 
    
    Default n_estimators = 50
    """
    forest_scores = [] # instantiate a list to keep track of cross-val scores
    time_split = TimeSeriesSplit(n_splits=5) # define a cross-val generator with 5 splits
    forest = RandomForestClassifier(n_estimators=n_estimators, n_jobs=3) # define a random forest with n_estimators
    
    for key, data in train_dict.items():
        this_score = np.mean(cross_val_score(forest, data, y_train,
                                                scoring='roc_auc', cv=time_split,
                                                n_jobs=3, verbose=2))
        forest_scores.append(np.round(this_score, 4))
        print('Finished training with {}'.format(key))
        
    return forest_scores

3.2. Logistic Regression¶

Logistic Regression comes with all of the benefits of a linear regression model but also includes a "decision" layer on top of its output that labels data between two classes. In this way, logistic regression can be quite fast thanks to the benefits of vectorized math, and it generally performs well when the data is linearly separable. However, this also means more complicated non-linear features need some handling before implementing a logistic approach. At its core, logistic regression wants to identify various separation parameters and look at the probability that a given point is on one side of the boundary or the other.

Knowing that our web session data has non-linear dependencies, we might hypothesize that the first data set performs worse than the other two as the model appears to fail in separating these features across a line/hyperplane. By including some interpretation of these non-linear features (whether it's through introducing polynomial features or distinct time-of-day categories), we give the model more defined classifications in the total feature space.

Let's see how our model handles the data:

3.3. Support Vector Machines (SVM)¶

On the topic of separating data through hyperplanes, this is where Support Vector Machines (SVM) shine. In simplest terms, an SVM wants to find the cleanest boundaries that keep similarly classified data points together. Where a logistic regression model uses hyperplanes to come up with probability density functions to indicate whether data should be classified as belonging to Column A or Column B, an SVM is more interested in maximizing the "no man's land" between separable data.

Similar to logistic regression, improving a dataset's linear separability will often result in better linear-SVM results; however, SVMs can also consider the data over higher-order feature spaces by utilizing different similarity functions known as kernels. These kernels are powerful functions that can describe both simple linear and non-linear similarities alike. As long as the SVM model knows which kernels to consider, complicated feature dependencies can be theoretically solved.

But with all good things, this comes at a price -- specifically, the computational cost. As a reminder, our datasets have approximately 250,000 examples and over 10,000 features. According to Scikit-learn's documentation, the SVMs scale with at least $O(n_{examples}\times n_{features}^{2})$. In fact, running the SVM classifier sklearn.svm.SVC() will cost us many times the computational cost of our random forest classifier so we will settle for the simpler linear-SVM model SGDClassifier().

Once again, we might expect that datasets that include linearly-separable features to perform better than those with non-linear dependencies. Let's see if we see score improvements between our three sets similar to those in the previous model.

3.4. XGBoost¶

Short for eXtreme Gradient Boosting, XGBoost is an open-source tool developed by Tianqi Chen that has become popular among garage data scientists and industry engineers alike. Known to be a fast and scalable technique, this tool combines the core aspects of random forests with the error-correcting power of gradient descent for each tree predictor. In this way, the tree classifiers are continually improved, and the final model can perform surprisingly well even with un-engineered data.

Let's see how this model performs:

Part 4. Conclusion¶

To wrap up, we looked at four different classification techniques to determine whether we could differentiate an authorized users web sessions from those of an intruder. Looking at three interpretations of the data: the first with tf-idf site name features and six time features; the second adding ten second-order time features; and the third instead adding two sets of time-of-day/year features, we found that these four algorithms each came with their advantages, but in the end there was a clear winner.
